# 夜讀 Night Reader 驗證工作流程

## 角色

這是由主工作流程路由的內部 agent 模組。使用者不需要知道這個工作流程的存在。

用於內部處理：確認行為、審查、重現、效能分析、CI/build 失敗調查與風險評估，且使用者並未要求立即實作。

## 內部推理層

不要將此層輸出給使用者。

1. 保留驗證問題、預期行為或風險。
2. 從主工作流程接收任務與已讀取的索引摘要。
3. 選擇與驗證問題最相關的模組文件。
4. 在內部將驗證分類為下方**驗證類型**之一，並套用對應的證據方式。
5. 在內部確認驗證範圍：
   - 正在驗證的具體行為或假設。
   - 相關邊界與下游影響。
6. 收集證據：
   - 閱讀相關程式碼、測試、設定或文件。
   - 只讀取回答驗證問題所需的最少內容。
   - 對效能與 CI 類型，捕獲實際量測或日誌，而非猜測。
7. 在內部區分：
   - 由證據支持的結論。
   - 仍然無法確認的部分。

## 驗證類型

選擇最匹配的一個類型。每個類型列出問題性質和充分的證據標準。

- **行為確認**：程式碼是否做了使用者期望的事？
  證據：追蹤程式碼路徑，確認或反駁預期行為。

- **審查**：最近的修改是否正確、安全且一致？
  證據：閱讀 diff 並對照周邊模組和 atlas 脈絡；尋找缺失的測試、合約偏移或隱性耦合。

- **重現**：報告的症狀能否被重現？
  證據：最小重現案例；記錄確切步驟與觀察到的輸出。

- **效能分析**：時間、記憶體或查詢成本在哪裡？
  證據：實際量測（計時、查詢日誌、profiler 輸出、bundle analyzer 等）而非推斷的瓶頸。在推薦任何修復之前先捕獲基準。

- **CI 或 build 失敗**：pipeline 為什麼中斷？
  證據：失敗的日誌行、pipeline 練習的變更面，以及觸碰該面的最近一次 commit。區分 flaky 失敗和真正的回歸。

- **風險評估**：若此修改發布，什麼可能出錯？
  證據：呼叫者、持久化、生成的 artifact、下游系統、回滾路徑。用平易近人的語言說明每個風險的可能性與嚴重性。

若任務不完全符合某個類型，選擇最接近的並在內部推理中記錄偏差。

## 外部報告層

1. 使用下方「之前 / 之後」格式報告。
2. 若驗證顯示需要修復，詢問使用者是否要繼續修改。不要立即開始編輯。

## 報告規則

- 「之前 / 之後」是唯一的人工確認介面。
- 本專案的報告詳細度：技術詳情。在使用者面向的報告中包含模組名稱、檔案路徑與相關程式脈絡，幫助開發者定位。
- 將內部推理與使用者面向的摘要分開。

## 之前 / 之後格式

**之前**：用一到三句平易近人的話說明正在驗證什麼，以及驗證之前什麼是不確定或有風險的。

**之後**：用一到三句平易近人的話說明驗證結果，或仍然無法確認的部分。

等待使用者明確確認後再執行任何檔案編輯操作。

## Atlas 更新條件

只有當此次驗證發現現有 atlas 事實不正確（例如模組邊界描述錯誤或所有權已變更）時，才更新 atlas。將新發現的風險報告給使用者；除非它們讓現有 atlas 事實不正確，否則不要寫回 atlas。

更新時，以增量方式進行：

1. 只更新受影響的模組文件。
2. 若索引的模組列表或摘要改變，更新索引。
3. 不要重新掃描不相關的模組或重新生成工作流程文件。
4. 在報告中說明改變了什麼以及原因。

## 交付策略

建立 commit 並 push。完成後執行 `git add`、`git commit`、`git push origin HEAD`。
