# 墨頁 Inkpage Reader 修改工作流程

## 角色

這是由主工作流程路由的內部 agent 模組。使用者不需要知道這個工作流程的存在。

用於內部處理所有程式碼修改任務：bug 修復、新功能、優化、重構、發布、依賴升級、schema/資料遷移、設定變更、hotfix 與清理。

## 內部推理層

不要將此層輸出給使用者。

1. 保留使用者的原始請求。
2. 從主工作流程接收任務與已讀取的索引摘要。
3. 選擇與任務最相關的模組文件。
4. 在內部將任務分類為下方**內部任務類型**之一，並套用對應的推理提示。
5. 在內部校準範圍，確保「之前 / 之後」準確：
   - 什麼會改變。
   - 下游可能受到什麼影響。
   - 哪些邊界仍不確定。
6. 確認是否觸發決策閘門（見下方）。若有任何觸發，改用決策閘門格式，而非直接呈現「之前 / 之後」。
7. 如果修改不簡單 — 涉及三個以上的檔案或跨越一個以上的模組 — 在編輯前先撰寫短篇修改計畫至 `docs/changes/<YYYY-MM-DD>-<slug>.md`（若 `docs/changes/` 不存在則建立）。計畫列出範圍、選定的模組入口、預期的檔案觸及範圍、驗證步驟與回滾備註。這個計畫是工程草稿，不是使用者面向的確認。
8. 只有在需要時才查閱程式碼、測試、文件、設定、runtime 路徑或參考資料。

## 內部任務類型

選擇最匹配的一個類型。每個類型列出工作性質、完成前的最低驗證期望，以及回滾方式。

- **Bug**：現有行為錯誤或不穩定。
  對於非簡單的 bug，在提出根本原因之前，使用下方**結構化 bug 調查工作流程**。
  驗證：重現消失；相關測試通過；最近的整合測試覆蓋已修復路徑。
  回滾：回退修補；bug 重現但其他行為不變。

### 結構化 Bug 調查工作流程

對任何非簡單的 bug，在提出根本原因之前使用此工作流程。目標是避免過早收斂到最近的症狀層。

1. 分類症狀：判斷是確定性的或間歇性的；判斷是 UI、狀態、持久化、async/時序、快取、lifecycle 或外部依賴相關；說明觸發 bug 的確切使用者操作或系統事件，以及最終的錯誤狀態。
2. 追蹤使用者操作鏈：從使用者可見的入口點開始，追蹤 callback、coordinator、service、repository、state machine、renderer 和持久化，直到找到預期的最終狀態。
3. 識別狀態所有者與目標狀態：說明擁有受影響狀態的物件或模組；識別應該改變的確切欄位、資料庫記錄、快取條目或 widget；識別能證明狀態正確改變的可觀察結果。
4. 列出受影響狀態的所有寫入者和恢復者：包括透過 listener、post-frame callback、timer、animation、async completion、lifecycle hook、observer、cache 和持久化恢復路徑的間接寫入者。
5. 列出 async 邊界和 invalidator：await 點、future、stream、post-frame callback、listener/notifier、timer、animation、request id、generation counter、stale-request guard、cancellation check、cache invalidation、retry 和 fallback 邏輯。
6. 比較候選根本原因：至少提出兩個可能原因；解釋各候選的支持與削弱點；優先選擇以最少假設解釋觸發時序、間歇性與觀察到的最終狀態的原因。
7. 設計重現測試：對確定性 bug，直接練習失敗路徑；對間歇性 bug，顯式交錯競爭的 async 操作。
8. 然後才提出修復：說明最小程式碼改動；解釋為什麼它解決根本原因而不是掩蓋症狀；識別回滾路徑和相關驗證指令。

- **功能**：請求新行為。
  驗證：新測試覆蓋 happy path 和至少一個失敗路徑；build 成功；type check 通過。
  回滾：功能 flag 或回退；現有行為不受影響。

- **優化**：行為不變但品質提升。
  驗證：現有測試仍通過；目標指標（延遲、記憶體、bundle 大小、查詢次數）在修改前後有量測。
  回滾：回退；品質退回先前的基準。

- **重構**：結構改變但預期行為不變。
  驗證：`flutter test` 通過；公開 API 未改變（除非明確納入範圍）。
  回滾：回退；行為任一方向均不變。

- **發布**：升版、準備 tag。
  驗證：`pubspec.yaml` 版本號與 changelog 條目匹配；`flutter analyze` 和 `flutter test` 通過；遵循發布規則（先推 branch 再建 tag）。
  回滾：刪除 tag 並回退版本 bump；release 未發布。

- **依賴**：升級、降級、替換或移除套件。
  驗證：`flutter pub get` 成功；build 成功；覆蓋受影響套件呼叫者的測試通過；檢查 changelog 是否有重大變更。
  回滾：固定回前一版本並重新解析。

- **遷移**：資料庫 schema、資料或儲存遷移。
  驗證：forward migration 在全新資料上成功；reverse migration script 存在或不可逆性已記錄；migration 後至少有一個讀取和一個寫入路徑被驗證。
  回滾：reverse migration script 或從快照還原。**始終升級至決策閘門。**

- **設定**：環境變數、功能 flag、基礎設施設定或 runtime 參數。
  驗證：設定在受影響環境中正確載入；相關程式碼路徑響應新值；secrets 未 commit。
  回滾：回退至先前設定值；重啟或重新載入受影響的流程。

- **Hotfix**：正在發生的 incident 的緊急修復。
  驗證：最小重現消失；已修補 branch 上其餘測試仍通過；記錄了一個後續任務用於正式修復或事後分析。
  回滾：回退 hotfix；incident 重現。優先選擇最小的修改面。

- **清理**：移除死碼、合併檔案、修剪註解或刪除未使用資源。
  驗證：build 成功；`flutter test` 通過；已移除符號沒有殘留引用；使用者面向行為不變。
  回滾：回退；沒有任何東西遺失，因為沒有行為改變。

若任務不完全符合某個類型，選擇最接近的並在內部推理中記錄偏差。不要發明新類型。

## 外部報告層

1. 若觸發了決策閘門，先呈現決策閘門格式，等待使用者選擇後再繼續。
2. 使用下方「之前 / 之後」格式與使用者確認。
3. 等待使用者明確確認後再編輯任何檔案。
4. 撰寫修改計畫並 commit（若需要）：
   - 建立 `docs/changes/<YYYY-MM-DD>-<slug>.md`，內容包含：標題（`# Change Plan: <slug>`）、日期、任務類型、Before、After、Scope（涉及檔案與模組）、Validation Steps、Rollback。
   - 儲存至 `docs/changes/<YYYY-MM-DD>-<slug>.md`。
   - `git add` 並 `git commit -m "plan: <slug>"`。
   - 在此 commit 成功之前不要編輯任何原始碼檔案。
5. 計畫 commit 後，實作修改。
6. 執行選定任務類型的最低驗證。在使用者面向的報告中包含驗證結果。若驗證失敗，不要聲稱完成；要麼修復並重新驗證，要麼誠實報告失敗並詢問使用者如何繼續。
7. 驗證受影響的行為與邊界。
8. 用一句平易近人的話描述改變了什麼以及如何驗證。

## 報告規則

- 「之前 / 之後」是唯一的人工確認介面。
- 本專案的報告詳細度：技術詳情。在使用者面向的報告中包含模組名稱、檔案路徑與相關程式脈絡，幫助開發者定位變更。
- 將內部推理與使用者面向的摘要分開。
- 驗證狀態是使用者面向報告的一部分，不論報告詳細度：直接說明檢查通過、跳過或失敗。

## 之前 / 之後格式

**之前**：用一到三句平易近人的話說明目前的狀況以及什麼是錯誤的、缺失的或有風險的。

**之後**：用一到三句平易近人的話說明修改完成後會成立什麼。

等待使用者明確確認後再進行任何檔案編輯操作。

## 決策閘門

決策閘門是在「之前 / 之後」的基礎上，針對影響範圍更廣的修改進行升級。當以下任何觸發條件符合時使用：

- 修改會改變模組邊界（建立、移除或合併模組）。
- 修改影響外部 API 合約或公開介面。
- 有兩種或多種可行方案，且各有不同的取捨。
- 修改涉及不可逆操作（大規模刪除、資料庫遷移、框架替換、有資料損失風險的套件降級）。
- 內部任務類型為「遷移」或包含不可逆步驟。

觸發時，呈現以下格式代替純粹的「之前 / 之後」：

```markdown
## 決策：<一句話標題>

### 背景
為什麼需要這個決策。

### 方案
A. <方案 A> — <優點 / 成本>
B. <方案 B> — <優點 / 成本>

### 影響
哪些區域受影響以及如何影響。

### 建議
推薦哪個方案及原因。
```

使用者選擇後，繼續以「之前 / 之後」實作所選方案。記錄決策：

- 跨模組決策：在索引的架構決策表中新增一行。
- 模組級別決策：在受影響模組的 Known Risks 或 Do Not Do 欄位新增備註。

## Atlas 更新條件

只有當修改確實改變了模組邊界、所有權或外部 API 時，才更新受影響的 atlas 文件。一般 bug 修復和小功能不需要 atlas 更新。

更新時，以增量方式進行：

1. 只更新受影響的模組文件。
2. 若索引的模組列表或摘要改變，更新索引。
3. 不要重新掃描不相關的模組或重新生成工作流程文件。
4. 在報告中說明改變了什麼以及原因。

## 交付策略

建立 commit 並 push。完成後執行 `git add`（列出具體檔案）、`git commit`、`git push origin HEAD`。對於發布任務，嚴格遵守 `AGENTS.md` 的發布規則：先推 branch，再建 tag，再推 tag。
