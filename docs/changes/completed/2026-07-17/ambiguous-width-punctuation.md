# 歧義寬度標點佔一格 — 彎引號/間隔號轉 CJK 專屬碼位

層級：T1（transformer 純函式層單點變更，可逆，診斷明確）

## Before（現況與根因）

使用者需求：標點符號要佔一格、與中文字同寬。

現況：`normalizeTypography`（預設開）已把 CJK 脈絡的半形 `,.!?;:` 轉為
全形碼位，全形標點由 CJK 字型繪製、佔一格，這部分沒問題。缺口在
**東亞歧義寬度（East Asian Ambiguous）字元**：

- 彎引號 `“ ” ‘ ’`（U+201C/201D/2018/2019）——網文極常見；
- 間隔號 `·`（U+00B7，譯名分隔）與 `‧`（U+2027）。

這些碼位 Roboto 有字形，Android/Flutter 字型回退鏈逐字取「第一個有
字形的字型」，永遠命中 Roboto 的西文窄字形，不會落到 Noto Sans CJK，
因此真機上不佔一格、與中文字不同寬。`fwid` font feature 只作用在
「半形→全形」有 OpenType 對映的字形，Roboto 不支援，救不了這批字元。
本機測試字型（FlutterTest）所有字形等寬，此症狀只在真機可見。

## After（改法與驗證）

文字層把歧義寬度標點轉成 **CJK 專屬碼位**（Roboto 沒有字形，回退鏈
必然落到 CJK 字型、佔滿一格），在 `normalizeTypography` 的
`normalizePunctuation` 傘下（預設開）：

1. **彎雙引號 `“…”` → `「…」`、彎單引號 `‘…’` → `『…』`**（繁中書名層級慣例）。
   - 採**成對掃描**而非逐字元轉換：`“` 找到對應 `”` 才轉，兩端一起轉，
     不會出現「一邊轉一邊不轉」的破對（逐字元 + 鄰字判斷會在
     `他說“Hello”` 這種案例破對）。
   - 轉換條件：引號對的**內文或外側鄰字**含 CJK 脈絡字元（漢字或 CJK
     標點/全形區段）。純英文段落的引號原樣保留。
   - 撇號保護：`’` 前後皆為拉丁字母/數字（`don’t`、`boys’`）不視為
     引號收尾。不成對（落單）的彎引號原樣保留。
2. **間隔號 `·`（U+00B7）/`‧`（U+2027）→ `・`（U+30FB）**，
   條件：前後皆為漢字（譯名場景，如 `哈利·波特`）。
3. 既有 `pairQuotes` 激進項（直引號 `"` 配對→「」）維持不變，兩者
   目標碼位一致（「」），風格統一。

明確不做（記錄理由）：

- `…`（U+2026）：`……` 正規化已存在，但 U+2026 本身仍由 Roboto 繪製
  （偏窄、貼底線）。無 CJK 專屬替代碼位，文字層無解；若真機目視
  仍不可接受，後續需走字型層方案（隨 App 附帶小型標點子集字型，
  另開 Decision Gate）。
- `—`（U+2014）：Roboto 的 em dash 本身即約 1em 寬，僅 `——` 相連處
  可能有細縫，非佔格問題，不動。

契約確認：變更僅發生在 transformer 階段（displayText 定形前，唯一
允許改字的位置）；contentHash 隨文字改變 → metrics 磁碟快取自然
冷啟重建（設計內路徑）；不涉樣式，StyleFingerprint 不變。

驗證：
1. `reader_v2_content_transformer_test.dart` 新增對照表單測：
   中文彎引號轉換、巢狀單引號、`他說“Hello”` 成對轉換、純英文引號
   不動、撇號不動、落單引號不動、間隔號轉換與不誤傷（`3·14` 不轉——
   數字非漢字）。
2. `flutter analyze`、`flutter test` 全綠。
3. 真機目視：彎引號/間隔號佔一格；順帶檢查 `……` 視覺是否可接受。
