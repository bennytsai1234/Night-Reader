# 夜讀 Night Reader Codebase Atlas

這是本專案日常工作的自足入口與路由器。它自帶紀律——沒有單獨的工作流程文檔需要閱讀。

## 進入點

1. 保留使用者的原始請求。
2. 閱讀 `night_reader_index.md` 一次，然後用一句白話確認這個專案是做什麼的。
3. 只從索引中挑選相關的模組文檔——永遠不要全部閱讀。如果對領域不熟悉，先縮小到模組地圖，再收窄。
4. 依意圖路由：**知道**（解釋、定位、可行性、擁有權、行為檢查、審查、重現、分析、CI 失敗、風險）→ 調查；**變更**（任何程式碼編輯）→ 變更；混合／不清楚 → 先調查，再決定。
5. 傳遞結論；不要跨步驟重複閱讀索引或模組文檔，除非需要尚未收集的上下文。

## 調查（唯讀）

從 atlas 加上最少必要程式碼來回答；將已確認的事實與假設和未知分開。永遠不要編輯——如果需要修復，在使用者同意後移交給變更。依問題需求應用紀律：除錯 = 重現 → 排序假設 → 二分定位；審查 = 對照擁有模組和邊界模組閱讀差異；開放設計問題 = 一次訪問一個問題，每個附帶建議答案。

## 變更（任何編輯）

判斷紀律層級，然後縮放投入：

- **T0 微小**（無邏輯變更、可逆、單一檔案）：一行 Before/After；跳過計劃檔案；執行最相關的一項檢查。
- **T1 一般**（封閉、可逆、診斷明確）：當有便宜接縫時加入一個聚焦測試；在編輯原始碼前寫入草稿計劃 `docs/changes/planning/{{DATE}}-{{SLUG}}.md`。
- **T2 困難／有風險**（非同步／有狀態的 bug、跨模組、外部 API、不可逆、效能迴歸、診斷不確定）：完整紀律；同樣的計劃檔案；通常需要決策閘。

**硬底線：** 不可逆、跨模組、外部 API 或遷移工作至少是 T2。尊重明確的「快一點／徹底一點」覆寫，但永遠不低於底線。

**Before / After 閘**（唯一的確認介面）——在編輯任何檔案前等待明確確認：
- **Before**：目前狀態和診斷出的根本原因，用白話說明。
- **After**：什麼會變成真，以及將如何驗證。

**決策閘**——當變更會改變模組邊界、外部 API、不可逆或遷移、或有兩種以上可行方案時：呈現背景／選項（A/B 附取捨）／建議，並在 Before/After 之前等待選擇。跨模組決策記錄在索引的架構決策表中；模組層級決策記錄在該模組的已知風險中。

編輯後，依層級縮放驗證；驗證結果總是在回報中——永遠不要在檢查失敗時宣稱完成。完成後，將計劃移至 `docs/changes/completed/{{DATE}}-{{SLUG}}.md`。只在模組邊界、擁有權或外部 API 變更時更新 atlas 文檔——增量更新，不重新掃描。

## 回報與交付

- 回報方式：純白話——使用者面向回報中不出現模組名稱、路徑或程式碼。
- 交付方式：commit 並 push
- 除非使用者明確要求完整重建，否則不要重新執行 Codebase Atlas 初始化。
