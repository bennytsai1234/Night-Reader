# 夜讀 Night Reader Change Workflow

## 角色

這是 adapter 入口步驟路由的內部 agent 模組，使用者不需要知道這個工作流程的存在。

用於所有程式碼修改任務：bug 修復、功能新增、優化、重構、發布、依賴升級、schema 或資料遷移、設定變更、hotfix 和清理。

此工作流程使用的技術文件（debugging、TDD、verification）以獨立文件的形式放在 `night_reader_techniques/` 下，按需讀取，永遠不內嵌於此。只讀取當前任務所需的技術文件。

## 紀律等級

每個任務開始時先判斷需要多少紀律。在高風險的地方多投入精力；不要對瑣碎的工作套用繁重的流程。

- **T0 — 瑣碎：** 無行為邏輯變更、可逆、單一檔案（typo、註解、常數、明顯安全的單行修改）。不需要 debugging/TDD。一行的之前/之後。跳過計畫檔。使用最相關的一個檢查來驗證。
- **T1 — 正常：** 範圍明確、可逆、診斷清楚。輕量路徑：修復已確認根本原因；如果存在合適的測試接縫，新增一個聚焦的測試。短計畫作為未 commit 的草稿。對應類型的測試子集。
- **T2 — 困難/高風險：** 間歇性/非同步/有狀態的 bug、跨模組、外部 API、不可逆操作、效能回歸，或尚不確定的診斷。完整的紀律；通常觸發決策門；完整的驗證。

**硬性下限：** 任何不可逆、跨多個模組、改變外部 API 合約或屬於遷移的變更，至少是 T2，無論大小。這些條件通常也會觸發決策門（不改變邊界的跨模組工作可能不會）。

**控制：** 自動判斷等級。如果使用者說「快一點」/ 「仔細做」（或類似說法），遵從該指示，但永遠不要低於硬性下限。

## 內部推理層

不要將這一層輸出給使用者。

1. 保留使用者的原始請求。
1. 從入口步驟接收任務與已讀取的索引摘要。
1. 選擇最相關的模組文件。
1. **判斷紀律等級**（T0/T1/T2），使用上方規則。
1. 在內部將任務分類為下列任務類型之一，並按等級讀取對應的技術文件：
   - **Bug** → 讀取並遵循 `night_reader_techniques/debugging.md`（T1 = 輕量路徑，T2 = 完整回饋迴圈與排名假設）。
   - **行為變更（功能/bug 修復）** 且存在廉價測試接縫 → 讀取並遵循 `night_reader_techniques/tdd.md`（T0 跳過）。
   - **重構** → 套用下方重構說明。
1. 校準範圍，使之前/之後準確：什麼會改變、下游可能受影響什麼、哪些邊界仍不確定。
1. 確認是否有任何決策門觸發條件（見下方）。如果有，在之前/之後之前使用決策門。

## 內部任務類型

恰好選一種類型。每種類型說明工作內容、聲稱完成前預期的最低驗證，以及回退路徑。驗證命令是針對本專案的（`flutter analyze`、`flutter test`）。

- **Bug**：目前行為錯誤或不穩定。在提出根本原因之前，通過 `night_reader_techniques/debugging.md` 調查；在 T2 時這是強制的。驗證：重現已消失；相關測試通過。回退：恢復修補程式。
- **Feature**：新行為。驗證：新測試涵蓋正常路徑和一個失敗路徑；`flutter analyze` 通過。回退：feature flag 或 revert。
- **Optimization**：相同行為，更好的品質。驗證：現有測試通過；目標指標在修改前後都有量測。回退：恢復到先前的基準。
- **Refactor**：結構改變，行為不變。驗證：`flutter test` 完整套件通過；公開 API 未變更（除非明確列入範圍）。回退：revert。
- **Release**：版本號更新、changelog、tag。驗證：`pubspec.yaml` 版本與 changelog 一致；`flutter analyze` 和 `flutter test` 通過；tag 遵循專案慣例（`vX.Y.Z`）。回退：刪除 tag 並恢復版本號更新。
- **Dependency**：升級/降級/替換/移除套件。驗證：`flutter pub get` 成功；`flutter analyze` 通過；涵蓋呼叫者的測試通過；確認 release notes 是否有破壞性變更。回退：固定到先前版本。
- **Migration**：schema、資料或儲存遷移。驗證：正向遷移在全新資料庫上成功；存在逆向腳本或記錄了不可逆性；遷移後至少有一個讀取路徑和一個寫入路徑正常運作。回退：逆向腳本或從快照還原。**始終升級到決策門。**
- **Config**：環境變數、feature flags、infra 設定、執行時參數。驗證：設定在受影響的環境中載入；依賴路徑有回應；沒有提交的 secrets。回退：恢復值；重新載入程序。
- **Hotfix**：對線上事故的緊急修復。驗證：最小重現已消失；修補的分支上套件仍通過；記錄了後續任務。回退：revert；盡量使修改面最小。
- **Cleanup**：死程式碼、整合、移除未使用的 asset。驗證：`flutter analyze` 和 `flutter test` 通過；沒有對已移除符號的殘留參照；行為不變。回退：revert。

如果任務不符合任何類型，選最接近的並在內部記錄偏差。不要發明新類型。

### 重構說明

重構時，目標是**深化**：更多行為在更小、更可測試的介面後面。對任何你懷疑是淺層傳遞的東西套用**刪除測試** — 想像刪除它：如果複雜性消失了，它沒有貢獻；如果複雜性出現在其呼叫者中，它是有價值的。改變結構，永遠不改變行為，並在整個過程中保持完整套件通過。

## 外部報告層

1. 如果有決策門觸發條件，先呈現決策門格式並等待使用者選擇。
1. 用之前 / 之後格式與使用者確認。
1. 等待使用者明確確認後才編輯任何檔案。
1. 按等級記錄計畫 — 這是工程草稿，不是使用者面向的確認。`{{DATE}}` 和 `{{SLUG}}` 在每次修改時填入；它們不是初始化佔位符：
   - **T0：** 跳過計畫檔。
   - **T1：** 寫入 `docs/changes/{{DATE}}-{{SLUG}}.md`（如需要建立 `docs/changes/`），包含這些欄位：任務類型、確認的之前、確認的之後、預期的檔案範圍、驗證步驟和回退路徑。留作未 commit 的草稿。在它存在之前不要編輯源碼檔案。
   - **T2：** 寫入相同的計畫檔。由於交付策略是 `no commit`，留作未 commit 的草稿。在計畫檔存在之前不要編輯源碼檔案。
1. 實作修改。
1. 讀取並遵循 `night_reader_techniques/verification.md` 進行驗證，按等級進行。在使用者面向的報告中包含驗證結果。如果驗證失敗，不要聲稱完成：修復並重新驗證，或誠實報告失敗並詢問如何繼續。
1. 以一句平易近人的話結束，描述改變了什麼以及如何驗證。

## 報告規則

- 之前 / 之後是唯一的人工確認介面。
- 本專案的報告詳細度：technical（技術詳情）
  - 技術詳情：在使用者面向的報告中包含模組名稱、檔案路徑和相關程式脈絡，幫助開發者定位修改。
- 將內部推理與使用者面向的摘要分開。
- 驗證狀態是使用者面向報告的一部分，無論報告詳細度如何：直接說明檢查是否通過、跳過或失敗。

## 之前 / 之後格式

**之前**：用一到三句平易近人的話說明目前的狀況與診斷出的問題 — 什麼是錯誤的、缺失的或有風險的，以及你對根本原因的判斷。使用者在這裡判斷你的診斷；如果它是淺層的或錯誤的，這是他們發現的地方。

**之後**：用一到三句平易近人的話說明修改完成後會成立什麼，以及如何驗證。

等待使用者明確確認後才進行任何檔案編輯操作。

## 決策門

決策門是針對影響更廣的修改的升級，超越之前/之後。當以下任何觸發條件符合時使用：

- 修改會改變模組邊界（建立、移除或合併模組）。
- 修改影響外部 API 合約或公開介面。
- 有兩種或多種可行方法，各有不同的取捨。
- 修改涉及不可逆操作（大規模刪除、資料庫遷移、框架替換、有資料損失的套件降級）。
- 內部任務類型是 Migration 或包含不可逆步驟。

當決策涉及多個相互依賴的未解決決策或不明確的需求時，先遵循 `night_reader_techniques/design-grilling.md` 解決它：一次一個問題，每個問題都有建議答案，盡可能讀取 atlas 和程式碼來回答。然後呈現下面的選項。

觸發時，呈現這個而不是普通的之前/之後：

```markdown
## 決策：<一句話標題>

### 背景
為什麼需要這個決策。

### 選項
A. <選項 A> — <優點 / 代價>
B. <選項 B> — <優點 / 代價>

### 影響
哪些區域受到影響以及如何影響。

### 建議
建議哪個選項以及原因。
```

使用者選擇後，繼續執行選定選項的之前/之後。記錄決策：

- 跨模組決策：在索引的架構決策表中新增一行。
- 模組級別決策：在受影響模組的 Known Risks 或 Do Not Do 部分新增說明。

## Atlas 更新條件

只有當修改真正改變了模組邊界、所有權或外部 API 時，才更新受影響的 atlas 文件。普通 bug 修復和小型功能不需要 atlas 更新。

需要更新時，採用增量更新：

1. 只更新受影響的模組文件。
2. 如果索引中的模組列表或摘要有變化，更新索引。
3. 不要重新掃描無關模組或重新生成工作流程文件。
4. 在報告中說明更改了什麼以及原因。

## 交付策略

no commit（只寫檔案，使用者自行 commit）
