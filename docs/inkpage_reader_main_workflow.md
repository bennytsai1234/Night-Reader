# 墨頁 Inkpage Reader 主工作流程

## 角色

這是這個專案日常操作的唯一入口。使用者不需要知道理解、修改、驗證工作流程的存在。Agent 根據任務意圖自動路由，並在需要時組合多個工作流程。

## 工作流程

1. 在執行任何操作之前，先開啟 `docs/inkpage_reader_index.md`，同時保留使用者的原始請求與意圖。
2. 用一句平易近人的話確認這個專案做什麼。
3. 根據任務意圖路由：
   - 理解、解釋、調查或可行性評估：使用理解工作流程。
   - 修改、bug 修復、新功能、優化、重構、發布、依賴升級、schema/資料遷移、設定變更、hotfix 或清理：使用修改工作流程，並選擇對應的內部任務類型。
   - 驗證、審查、安全性確認、重現、效能分析、CI/build 失敗調查或風險評估：使用驗證工作流程。
   - 混合意圖：依理解 → 驗證 → 修改的順序組合工作流程。
4. 在內部執行對應工作流程。
5. 以平易近人的語言向使用者報告結果。
6. 如果任務涉及編輯檔案，提供「之前 / 之後」確認並等待明確確認後再編輯。
7. 依此交付策略完成：建立 commit 並 push。完成後執行 `git add`、`git commit`、`git push origin HEAD`。

## 路由原則

當意圖不明確時，先用理解工作流程，再決定是否需要驗證或修改。永遠不要在沒有「之前 / 之後」確認的情況下編輯檔案。

組合工作流程時，將前一個工作流程的結論帶入下一個。除非下一個工作流程需要前面未收集到的脈絡，否則不要重新閱讀索引或模組文件。

日常維護任務路由至修改工作流程，並指定明確的內部任務類型。

## 報告規則

- 「之前 / 之後」是唯一的人工確認介面。
- 本專案的報告詳細度：技術詳情。在使用者面向的報告中包含模組名稱、檔案路徑與相關程式脈絡，幫助開發者定位變更。
- 將內部推理與使用者面向的摘要分開。

## 之前 / 之後格式

**之前**：用一到三句平易近人的話說明目前的狀況以及什麼是錯誤的、不清楚的、缺失的或有風險的。

**之後**：用一到三句平易近人的話說明操作完成後會成立什麼。

等待使用者明確確認後再執行。

## 持續工作模式

每個任務完成後，用平易近人的語言詢問使用者：

```
還有其他需要處理的事嗎？
```

如果使用者繼續提出請求：

- 不要重新閱讀索引。
- 繼續進行路由判斷。
- 保持平易近人的報告風格與「之前 / 之後」機制。

只有當使用者說沒有其他事情需要處理時才結束。
